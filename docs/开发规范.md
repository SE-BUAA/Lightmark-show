# 开发规范 v1.0

建议是新建文件而不是修改旧文件，除非是涉及到自己特有部分的api修改或添加，merge时注意对比，不要把数据库配置喂给ai

## 一、版本控制规范（Git）

### 1.1 分支策略

- **`main`**：生产环境代码，只接受 `release/*` 或 `hotfix/*` 分支的合并。
    
- **`develop`**：主开发分支，所有功能分支从此拉出，并合并回此处。
    
- **`feature/{your_name}`**：新功能分支，从 `develop` 拉出，完成后合并回 `develop`。
    
- **`bugfix/{your_name}`**：修复非紧急 bug，从 `develop` 拉出，合并回 `develop`。
    
- **`release/vX.Y.Z`**：发布准备分支，从 `develop` 拉出，只做测试、版本号更新、文档整理，完成后合并到 `main` 并打 tag。
    

examle: feature/jyx_1

### 1.2 提交信息格式

采用 **Conventional Commits** 规范：

```Plaintext
<type>(<scope>): <subject>

[optional body]

[optional footer]
```

**type** 类型：

- `feat`: 新功能
    
- `fix`: Bug 修复
    
- `docs`: 文档更新
    
- `style`: 代码格式（不影响逻辑）
    
- `refactor`: 重构
    
- `perf`: 性能优化
    
- `test`: 测试相关
    
- `chore`: 构建/工具变动
    

### 1.3 新建文件原则

- **新增任何****源代码****、配置文件、测试文件、文档等，都必须通过 Git 跟踪，并随提交推送到远程仓库。**
    
- 禁止在服务器上直接修改文件而不提交；所有变更必须先提交到仓库，再通过部署脚本拉取。
    
- 临时文件（如日志、本地环境配置、IDE 元数据）必须添加到 `.gitignore`。

## 二、后端开发规范（Spring Boot）

### 2.1 包结构与命名

```Plaintext
top.ortus.lightmark
├── controller
├── service (接口)
├── service/impl (实现)
├── mapper
├── entity
├── dto (请求/响应)
├── config
├── utils
├── exception
└── common
```

### 2.2 新建文件规则

- **每个 Controller、Service、Mapper、Entity、DTO 都必须新建独立的** **`.java`** **文件。**
    
- 不允许在一个文件中定义多个顶层类（内部类除外）。
    
- 文件名采用 **大驼峰**（PascalCase），例如 `FlightSearchController.java`。
    

### 2.3 代码风格

- 使用 **Lombok** 减少样板代码（`@Data`, `@Builder`, `@Slf4j`）。
    
- 方法参数超过 3 个时，封装为 DTO 或使用 `@RequestParam` 分开。
    
- **Controller 层**：只做参数校验、调用 Service、返回统一响应。不写业务逻辑。
    
- **Service 层**：业务逻辑、事务管理。每个 Service 必须定义接口（便于替换和测试）。
    
- **Mapper 层**：继承 `MyBatis-Plus BaseMapper`，复杂查询使用 XML 或注解。
    

### 2.4 API 统一响应格式

```Java
public class ApiResponse<T> {
    private int code;     // 0 成功，非0失败
    private String msg;
    private T data;
}
```

所有 Controller 方法返回 `ApiResponse<?>`。

### 2.5 异常处理

- 自定义业务异常 `ApiException`，携带错误码和消息。
    

### 2.6 配置管理

- 使用 Spring Profiles：`application-dev.yml`（本地开发）、`application-prod.yml`（生产）。
    
- 敏感信息（数据库密码、AI API Key）通过环境变量注入，不允许硬编码。
    
- 所有配置项按模块分组，并添加注释。
    

### 2.7 AI 功能开发约定

- 所有 AI 相关接口统一放在 `controller.ai` 包下，例如 `AIController`。
    
- AI 调用逻辑封装在 `service.AIService` 中，Controller 只负责参数转换和响应。
    
- 每个 AI 接口必须配置超时（`RestTemplate` 或 `WebClient` 设置 readTimeout）。
    
- 记录请求耗时和输入 prompt（注意隐私脱敏），用于后期优化。

## 三、前端开发规范（Vue 3）

### 3.1 目录结构

```Plaintext
src/
├── api/                 # 按模块封装的 API 请求函数
│   ├── auth.js
│   ├── flight.js
│   └── ai.js
├── assets/              # 静态资源（图片、字体）
├── components/          # 公共组件
│   ├── layout/
│   └── common/
├── composables/         # 组合式函数（复用逻辑）
├── router/              # 路由配置
├── stores/              # Pinia 状态管理
├── views/               # 页面组件
│   ├── home/
│   ├── flight/
│   └── ...
├── utils/               # 工具函数
├── App.vue
└── main.js
```

### 3.2 新建文件规则

- **每个 Vue 组件（****`.vue`****）、每个 API 模块（****`.js`****）、每个 Store 模块（****`.js`****）都必须新建独立文件。**
    
- 组件文件名采用 **大驼峰**，例如 `FlightSearchForm.vue`。
    
- API 模块文件名采用 **小驼峰**，例如 `flightApi.js`。
    
- 组合式函数文件名以 `use` 开头，例如 `useAuth.js`。
    

### 3.3 代码风格

- 使用 Vue 3 **组合式 API**（`<script setup>`）。
    
- 组件模板中避免复杂表达式，复杂逻辑应抽取到 `computed` 或 `methods`。
    
- 样式使用 `scoped` 避免污染，优先使用 CSS 变量或 Tailwind（可选）。
    
- 所有 API 请求通过 `axios` 拦截器统一处理 token 和错误。
    

### 3.4 状态管理

- 使用 **Pinia**，按模块划分 store（例如 `useUserStore`, `useOrderStore`）。
    
- 避免在组件中直接修改 store 中的状态，使用 actions 修改。
    

### 3.5 路由权限

- 在 `router.beforeEach` 中根据用户 roles 动态添加路由或做跳转拦截。
    
- 需要登录的页面统一使用元信息 `meta: { requiresAuth: true }`。

## 四、数据库规范

### 4.1 脚本管理

- 对数据库表项的添加与删减及时同步到文档里。
    
- 禁止直接在服务器上执行 ALTER TABLE 而不更新脚本。
    

## 五、文档规范

### 5.1 必须提供的文档

- `README.md`：项目介绍、快速启动、环境要求。
    
- `API.md`：所有接口的详细说明（请求/响应示例）。
    

### 5.2 代码注释

- 类/接口：说明职责。
    
- 复杂方法：说明参数、返回、异常情况。

## 六、开发流程与协作

### 6.1 新功能开发步骤

1. 从 `develop` 拉取最新代码，创建 `feature/xxx` 分支。
    
2. 在本地按规范新建文件（Controller、Service、Mapper、DTO、Vue 组件等）。
    
3. 编写代码并自测（至少包含正向用例）。
    
4. 提交前运行 `mvn clean test`（后端）和 `npm run lint`（前端），确保无报错。
    
5. 提交信息遵循 Conventional Commits。
    
6. 推送到远程仓库，发起 Pull Request 到 `develop`。
    
7. 至少一名其他成员审核通过后合并。
    

### 6.2 代码审核要点

- 是否新建了必要文件？
    
- 是否有重复代码？
    
- 是否处理了异常和空值？
    
- API 响应是否符合统一格式？
    
- 前端组件是否拆分合理？
    
- 是否有提交敏感信息？
    

### 6.3 环境同步

- 建议开发时每个开发者本地使用本地数据库。
    
- 公共测试环境使用列出的公有数据库，从仓库模板复制并修改。

## 七、AI 功能开发特殊约定

由于 AI 接口较多且相互独立，建议：

1. **新建独立的 Controller**：`AIController.java` 统一处理所有 `/api/ai/*` 接口。
    
2. **每个 AI 能力对应一个 Service 方法**：例如 `generatePost()`, `sentimentAnalysis()`。
    
3. **提示词模板**：放在 `resources/prompts/` 目录下的 `.txt` 或 `.json` 文件中，便于管理和调优。
    
4. **测试**：为每个 AI 接口编写单元测试（Mock 外部 API），确保解析逻辑正确。
    
5. **容错**：当 AI 服务不可用时，返回降级提示（例如“AI 服务繁忙，请稍后再试”），而不是直接抛错。

## 八、禁止事项

- ❌ 禁止在代码中硬编码任何密码、token、API Key。
    
- ❌ 禁止在服务器上直接修改代码或配置而不提交到 Git。
    
- ❌ 禁止在一个文件中堆积多个不相关的功能。
    
- ❌ 禁止忽略提交前的代码检查和单元测试。
