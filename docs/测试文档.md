# 测试报告

## 1 引言

### 1.1 目标

本测试报告用于说明“拾光旅行 Lightmark”系统的测试范围、测试方法、测试环境、测试用例设计和测试执行结果。测试目标是验证系统在身份认证、用户中心、机票、酒店、火车票、度假产品、智能行程、社区、AI 助手和管理后台等核心业务中满足需求文档与详细设计说明书要求，并能够在典型用户流程下稳定运行。

### 1.2 背景

Lightmark 是一个前后端分离的一站式旅游服务平台。前端使用 Vue 3、TypeScript、Element Plus、Pinia 和 Vue Router，后端使用 Spring Boot 3.5、MyBatis-Plus、JWT、MySQL，并在测试环境中使用 H2 模拟 MySQL。系统支持普通用户和管理员两类角色，提供产品查询、下单支付、退票改签、订单管理、AI 文案生成、智能行程建议、游记社区和后台运营管理等能力。

### 1.3 范围

本次测试覆盖以下模块：

- 身份认证与权限控制：验证码、邮箱验证码、注册、登录、JWT 鉴权、管理员权限。
- 用户中心：个人资料、头像、密码、常用出行人、积分、会员等级、我的订单。
- 机票预订：搜索、价格日历、详情、下单预览、下单。
- 酒店预订：酒店列表、房型、订单、支付、取消、发票、评价。
- 火车票预订：站点选项、直达查询、中转查询、座位筛选、学生票/儿童票、支付、超时取消、退票、改签。
- 度假产品：目的地/出发城市/主题筛选、详情、AI 文案、下单、取消险、支付、退票、智能行程助手。
- 智能行程与社区：AI 生成行程、保存行程、游记、点赞、评论、问答、权限校验。
- AI 与对话：普通对话、区域补全、流式响应、AI 工具能力、度假详情和助手生成。
- 管理后台：仪表盘、用户管理、产品管理、订单管理、日志、问题回答、评论审核。

## 2 测试计划

### 2.1 目的

测试活动用于确认系统功能正确性、接口兼容性、数据一致性、异常处理能力和主要业务流程的可用性。对高风险流程，例如订单生成、支付状态变更、库存扣减、退票退款、火车票改签、AI 外部服务调用等，采用单元测试、集成测试和人工端到端测试结合的方式验证。

### 2.2 详细说明

#### 2.2.1 测试计划标识符

测试计划编号：Lightmark-TP-2026-001

#### 2.2.2 测试项

- 认证模块（AuthController、AuthService）
- 用户模块（UserController、UserService）
- 产品与订单模块（PublicApiController、OrderController、OrderService）
- 火车票模块（TrainController、TrainService、TrainMcpClient）
- 度假模块（VacationController、VacationService）
- 酒店模块（HotelController、HotelService）
- 智能行程与社区模块（ItineraryService、CommunityService）
- AI 对话模块（ConversationController、AIService、DeepSeekNativeChatService）
- 管理后台模块（AdminController、AdminService）

#### 2.2.3 要测试的特征

- REST API 是否按统一响应结构返回 `code`、`msg`、`data`。
- 认证失败、权限不足、参数错误、外部服务异常时是否给出可预期错误。
- 前端筛选条件是否与后端接口参数一致。
- 订单金额、优惠、退款金额、改签差价是否计算正确。
- 火车票直达与中转查询是否正确解析 12306 MCP 返回数据。
- 度假产品筛选项是否从 `product.extra` 与 `category_tags` 正确提取。
- AI 生成接口失败时是否不会阻断基础业务流程。
- 管理后台对用户、产品、订单的变更是否落库并可查询。

#### 2.2.4 不测试的特征

- 第三方真实支付扣款流程，仅测试模拟支付和订单状态流转。
- 真实短信服务，仅测试邮箱验证码和接口校验逻辑。
- 12306 外部 MCP 服务本身的正确性，仅测试本系统调用、解析、降级和错误提示。
- 大模型输出内容的事实准确性，仅测试接口可用性、响应格式和失败处理。

### 2.3 测试方法

- 单元测试：使用 JUnit 5、Spring Boot Test、Mockito 或 MockMvc，覆盖工具类、服务层规则和控制器错误处理。
- 集成测试：使用 H2 的 MySQL 兼容模式加载 `schema-h2.sql` 和 `data-h2.sql`，通过 MockMvc 调用真实 Controller。
- 接口测试：使用 Apifox/Postman 或浏览器 Network 调用 `/api/*` 接口，验证状态码、响应体和数据库副作用。
- 前端功能测试：在 Vue 页面中按用户流程手工执行，观察 Element Plus 弹窗、表格、分页、错误提示和 Pinia 状态同步。
- 回归测试：对历史缺陷进行复测，包括订单字段缺失、改签重复提示、火车票筛选无结果、度假行程字段转义异常等。
- 构建测试：后端执行 `mvn test`，前端执行 `npm run build`。

### 2.4 测试通过准则

- 后端单元测试和集成测试全部通过。
- 前端构建通过，无 TypeScript 编译错误。
- 关键业务流程均可完成：注册/登录、查询产品、下单、支付、取消/退票、订单查询。
- 数据库核心表字段满足业务需要，订单创建不出现非空字段缺失错误。
- 异常场景返回明确提示，不出现未处理异常堆栈暴露到前端。
- 管理员权限接口对普通用户拒绝访问。

### 2.5 测试任务

- 准备 H2 测试库和 MySQL 本地/远程测试库。
- 准备演示用户、管理员、产品、订单、游记、问题等基础数据。
- 执行后端自动化测试。
- 执行前端构建测试。
- 使用浏览器完成人工端到端测试。
- 记录缺陷、修复后执行回归测试。
- 更新测试文档与测试日志。

## 3 测试设计说明

### 3.1 目的

测试设计以业务流程为主线，将每个模块拆分为可验证的输入、处理和输出，并对订单、权限、库存、退款、AI 外部服务调用等高风险点设计边界和异常用例。

### 3.2 详细说明

#### 3.2.1 测试设计说明标识符

测试设计编号：Lightmark-TD-2026-001

#### 3.2.2 模块测试设计

- 认证模块：验证码缺失、弱密码、不支持邮箱域名、SMTP 未配置、正常登录。
- 用户中心：当前用户查询、资料更新、头像上传、密码修改、出行人增删改查。
- 火车票模块：站点列表、直达查询、中转查询、日期候选栏、列车类型筛选、座位筛选、购票、学生/儿童票、退票、改签一次限制。
- 度假模块：筛选项加载、POST 查询、详情查看、AI 文案生成、取消险价格、支付后显示取票码、智能行程助手、退票金额展示。
- 订单模块：待支付、已支付、已取消、退款，支付截止时间超时取消幂等性。
- 社区模块：游记列表、发布、点赞、评论、删除权限、问题回答权限。
- 管理后台：用户状态、会员等级、产品上下架/调价/调库存、订单状态干预、退款、日志查询。

#### 3.2.3 接口级测试重点

| 模块 | 接口示例 | 测试重点 |
| --- | --- | --- |
| 健康检查 | `GET /api/health` | 服务是否启动，统一响应是否正常 |
| 认证 | `POST /api/auth/login` | 图形验证码、密码校验、JWT 返回 |
| 用户 | `GET /api/user/current` | token 解析和用户信息返回 |
| 火车票 | `POST /api/trains/search` | 直达车次解析、筛选、座位与价格字段 |
| 火车票 | `POST /api/trains/transfer/search` | 中转两段车次、换乘时间、筛选规则 |
| 火车订单 | `POST /api/orders/train` | 座位选择、学生/儿童票价格、订单生成 |
| 火车退改 | `POST /api/orders/train/refund`、`POST /api/orders/train/change` | 取票码、退款金额、一次改签限制 |
| 度假 | `POST /api/vacations/search` | 目的地、出发城市、日期、主题筛选 |
| 度假 AI | `GET /api/vacations/{productId}/detail-ai` | AI 文案返回、失败提示 |
| 度假助手 | `GET /api/orders/vacation/{orderNo}/assistant` | 天气、穿衣、美食建议生成 |
| 社区 | `POST /api/posts/{id}/like` | 点赞状态切换 |
| 管理后台 | `PUT /api/admin/products/{id}/status` | 管理员权限和数据变更 |

#### 3.2.4 数据一致性设计

- `orders.total_amount`、`orders.pay_amount` 创建时必须有值。
- 火车票库存以余票字段计算，前端展示不直接使用原始库存。
- 学生票按原价 0.6 计算，非学生且年龄小于 18 岁按原价 0.8 计算。
- 退票金额按发车时间判断：15 天以上全额退还，15 天以内退还 50%。
- 改签后原订单和原取票码不可用，新订单标记不可再次改签。
- 度假订单购买取消险后应写入订单扩展信息并影响应付金额。

### 4.5 输入说明

- 正常输入：合法邮箱、强密码、真实日期、有效站点、有效座位、存在的产品 ID、有效取票码。
- 异常输入：空验证码、弱密码、非法邮箱域名、空站点、过期订单、重复改签、无效取票码、不存在产品 ID。
- 边界输入：库存为 0、票价为 0、年龄 17/18、退票时间刚好 15 天、支付截止时间刚过期。

### 4.6 输出说明

- API 输出：HTTP 状态码、统一响应体、业务数据字段、错误信息。
- 数据库输出：订单状态、支付金额、退款金额、取票码、库存或扩展信息变化。
- 前端输出：Element Plus 表格、分页、弹窗、确认框、错误提示、加载状态。

## 5 测试规程说明

### 5.1 目的

明确测试执行顺序，保证自动化测试、接口测试和人工回归测试可以重复执行。

### 5.2 测试规程标识符

测试规程编号：Lightmark-TPR-2026-001

### 5.3 特殊要求

- 自动化测试使用 `test` profile，避免污染真实 MySQL 数据库。
- 涉及 12306 MCP 和 AI 的测试需要可用网络和有效 API 配置；不可用时需验证前端友好错误提示和后端日志。
- 订单类测试需要使用独立测试数据，避免重复使用已退款或已改签的取票码。
- 每次数据库结构变更后，需要执行迁移/兼容性脚本并重新跑订单相关用例。

### 5.4 规程步骤

1. 启动 MySQL 或确认 H2 测试配置可用。
2. 后端执行：

```bash
cd backend
mvn test
```

3. 前端执行：

```bash
cd frontend
npm run build
```

4. 启动后端和前端开发服务器。
5. 使用浏览器执行核心端到端流程。
6. 使用 Apifox/Postman 验证关键 REST API。
7. 对失败用例记录缺陷、定位原因、修复后回归。
8. 更新测试日志和测试总结。
